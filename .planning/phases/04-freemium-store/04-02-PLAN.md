       `LICENSE_KEYS: KVNamespace;`
       alongside the existing ANTHROPIC_API_KEY field.

    2. Update CORS_HEADERS to also allow the /validate-key route:
       Add 'PUT' to Access-Control-Allow-Methods — or leave as 'POST, OPTIONS' since /validate-key uses POST. No change needed to CORS_HEADERS.

    3. In the fetch handler, between the OPTIONS check and the existing 'Method Not Allowed' guard (which currently only blocks non-POST), add the /validate-key route BEFORE the existing rename logic block:
       ```
       const url = new URL(request.url);

       if (request.method === 'POST' && url.pathname === '/validate-key') {
         try {
           const body = await request.json<{ key?: string }>();
           const key = typeof body.key === 'string' ? body.key.trim() : '';
           if (!key) {
             return new Response(JSON.stringify({ valid: false }), {
               status: 400, headers: { ...CORS_HEADERS, 'Content-Type': 'application/json' }
             });
           }
           const value = await env.LICENSE_KEYS.get(key);
           return new Response(JSON.stringify({ valid: value !== null }), {
             headers: { ...CORS_HEADERS, 'Content-Type': 'application/json' }
           });
         } catch {
           return new Response(JSON.stringify({ valid: false }), {
             status: 500, headers: { ...CORS_HEADERS, 'Content-Type': 'application/json' }
           });
         }
       }
       ```

    4. The existing `if (request.method !== 'POST')` guard and the rename try/catch block below it remain unchanged. The url.pathname check means /validate-key is handled before the existing POST logic runs. The existing route handles POST / (no pathname check needed since it's the fallback after /validate-key is checked).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Options page → Worker /validate-key | POST from extension to Worker; user-supplied key string crosses this boundary |
| Worker → Cloudflare KV | KV lookup is server-side; no secrets exposed to client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | /validate-key key string | mitigate | Worker trims and validates key is non-empty string before KV lookup; empty/non-string input returns { valid: false } with 400 |
| T-04-06 | Information Disclosure | KV key values | accept | KV stores only "active" as the value; key string itself is the secret; not exposed in Worker response |
| T-04-07 | Denial of Service | /validate-key endpoint | accept | Cloudflare Workers have built-in rate limiting per account; custom rate limiting deferred to v2 |
| T-04-08 | Spoofing | License key brute-force | accept | Key space is Trevor-defined; no account lockout in v1; brute-force risk accepted at $2.99 price point |
| T-04-SC | Tampering | npm/wrangler installs | accept | wrangler already a devDependency from prior phases; no new packages added |
</threat_model>
