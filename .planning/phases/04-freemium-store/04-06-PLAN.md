<objective>
Close the one genuinely-still-open finding from the Codex cross-AI review (04-REVIEWS.md MEDIUM: "chrome.notifications.onButtonClicked should be registered defensively... guard the notification ID or button index so unrelated future notifications do not open the upgrade URL accidentally").

Current code at `entrypoints/background.ts:57-61`:
```
chrome.notifications.onButtonClicked.addListener((notifId, btnIdx) => {
  if (btnIdx === 0) {
    chrome.tabs.create({ url: UPGRADE_URL });
  }
});
```
This only checks `btnIdx === 0`, not `notifId`. Any future notification (not just `'limitReached'`) with a button at index 0 would incorrectly open the upgrade URL. The fix is a one-line condition change plus a regression test — this was verified against the live codebase by the orchestrator before this plan was written; all other reviewer concerns (monthlyCount increment ordering, calendar-accurate reset, UPGRADE_URL centralization, Worker response codes) are already resolved in code and require no further action.

Purpose: Defensive correctness for a listener that is registered once and lives for the life of the extension — a single notification ID typo or a future second notification type would otherwise silently misroute the button click.

Output: entrypoints/background.ts gains a notification ID check. tests/background.test.ts gains coverage for both the matching and non-matching cases.
